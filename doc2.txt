云平台综合实训 · 拓展方向汇总
 
📋 总览
 
本拓展文档为已达到云平台综合实训基础要求的学生提供四个可选进阶方向。四个方向分别覆盖Kubernetes技能树
的不同分支，学生可根据自身兴趣和环境条件选择其中任意方向完成。
📌 方向选择建议
 
方向一 适合对基础设施感兴趣、想深入理解Kubernetes底层架构的学生。需要额外准备VMware虚拟化环境，与
手册主体（Docker Desktop）的环境不同，建议作为独立分支完成。
方向二 适合想掌握有状态应用部署、持久化存储等运维技能的学生。基于本手册的Docker Desktop环境，可与基
础部署无缝衔接。
方向三 适合对业务功能拓展、用户认证机制感兴趣的学生。需先完成方向二，在已有数据库基础上进行功能迭
代。
方向四 适合已有自己独立开发项目（如课程设计、毕业设计、个人作品）的学生，将项目容器化并部署到
Kubernetes集群，实现“自己的项目上云”。
⭐选用班级留言板项目可以自选方向一、二、三进行项目拓展和完善
⭐选用方向四（自己开发的项目）可以不用完成班级留言板项目，直接用自己的项目完成整个实训要求即可

方向一：从零搭建3节点Kubernetes集
群
 
📋 任务概述
 
在VMware或VirtualBox中创建3台Linux虚拟机，从零搭建Kubernetes集群，并完成班级留言板的完整部署与验证。
最终目标：掌握生产级Kubernetes集群的搭建与应用部署全流程能力。
🎯 任务要求
 
在VMware或VirtualBox中创建3台Ubuntu Server 22.04虚拟机（1 Master + 2 Worker），配置2核CPU、2GB内存、
20GB硬盘，设置静态IP并配置/etc/hosts实现主机名解析。所有节点关闭防火墙、关闭Swap、同步时间，安装
containerd（配置SystemdCgroup=true），安装kubeadm、kubelet、kubectl（v1.29）。Master节点执行kubeadm init初
始化集群（pod-network-cidr=10.244.0.0/16），安装Flannel网络插件，Worker节点执行kubeadm join加入集群，最终
所有节点状态均为Ready。
将班级留言板项目文件复制到Master节点，构建前后端Docker镜像（message-api:v1、message-frontend:v1等镜像），
确保镜像在集群所有节点可用。创建Namespace message-board ，部署Deployment （副本数2 ）、Service
（NodePort ）、ConfigMap ，验证所有Pod 状态为Running 。通过NodePort 访问前端页面（http:// 任意节点
IP:NodePort），验证健康检查接口（http://任意节点IP:30080/health）返回JSON，提交至少3条留言并确认留言
列表正常显示。

方向二：接入MySQL数据库
 
📋 任务概述
 
在前序任务中，班级留言板的后端服务（message-api）将留言数据存储在Node.js进程的内存中。一旦Pod因故障重
启、滚动更新或手动删除，所有留言数据将永久丢失。
本次任务要求你在已有Kubernetes集群中部署MySQL有状态服务，并改造后端应用，将存储引擎从内存切换为
MySQL数据库，实现留言数据的持久化存储。
最终目标：掌握有状态应用（StatefulSet）在Kubernetes集群中的部署方法，理解PersistentVolumeClaim持久化存储与
服务间依赖治理的运维实践。
🎯 任务要求
 
在已完成班级留言板基础部署（前端 + 后端，2个Deployment、2个Service、1个ConfigMap）的Kubernetes集群中，
完成以下操作：
数据库资源部署：创建Secret存储MySQL密码（Base64编码），创建PVC申请1Gi存储空间，编写init.sql初始化脚本
（创建messages表，字段：id自增主键、name、content、created_at，并插入一条默认留言"MySQL数据库已成功接
入！"），使用StatefulSet部署MySQL 5.7（副本数1），挂载PVC到/var/lib/mysql，挂载初始化脚本到/docker-
entrypoint-initdb.d，创建Headless Service（clusterIP: None）供StatefulSet使用。
后端应用改造：在backend项目中安装mysql2驱动，创建数据库连接池模块（db.js），改造POST /messages接口为
INSERT语句，改造GET /messages接口为SELECT查询，在应用启动时执行SELECT 1检测数据库连通性（连接失败
则进程退出，触发K8s重启），重新构建镜像并标记为message-api:v2。
应用部署更新：修改api-deployment.yaml 添加数据库环境变量（DB_HOST=message-db-service 、
DB_USER=app_user、DB_PASSWORD从Secret引用、DB_NAME=messages_db），为API容器添加startupProbe启动
探针（failureThreshold: 30，periodSeconds: 5）防止数据库启动慢导致Pod被误判为CrashLoopBackOff，按顺序先部
署数据库资源再更新API Deployment。
数据持久化验证：通过前端提交3条留言，手动删除message-db-0 Pod（模拟故障），等待StatefulSet自动重建后刷新
前端页面，确认所有留言数据完整保留。

方向三：用户认证与权限管理
 
📋 任务概述
 
在前序任务中，班级留言板允许任何用户匿名提交留言，无任何身份验证机制。这在实际生产环境中存在安全风险
（如恶意刷屏、冒充他人发言等）。
本次拓展任务要求你在现有留言板基础上，增加用户注册、登录和JWT身份认证功能，实现留言与用户的绑定，并
区分普通用户与管理员权限。
最终目标：掌握Web应用中的用户认证机制（JWT）、权限分级管理，以及前后端分离架构下的身份验证实现。
🎯 任务要求
 
在已完成班级留言板基础部署（前端 + 后端 + MySQL数据库）的Kubernetes集群中，完成以下功能拓展：
一、后端功能拓展
1. 用户数据模型：在MySQL数据库中新增 users 表，字段包括：id（自增主键）、username（唯一）、
password（需加密存储，使用bcrypt）、role（枚举类型：user / admin，默认 user）、created_at（时间
戳）。
2. 注册接口：新增 POST /api/auth/register 接口，接收 username 和 password，校验用户名是否已存在，密
码需加密后存入数据库。
3. 登录接口：新增 POST /api/auth/login 接口，验证用户名和密码，校验通过后签发 JWT Token（有效期24小
时），返回 token 和用户信息。
4. 认证中间件：编写JWT验证中间件，对需要登录的接口进行拦截校验（Authorization: Bearer <token>）。
5. 留言关联用户：改造 POST /api/messages 接口，在认证状态下自动将留言与当前登录用户绑定，在 
messages 表中新增 user_id 外键字段，关联 users 表。留言返回时携带留言人用户名。
6. 管理员权限：新增 DELETE /api/messages/:id 接口，要求只有 role=admin 的用户才能删除留言，普通用户
无权限。
7. 用户信息接口：新增 GET /api/auth/me 接口，返回当前登录用户的信息（用户名、角色）。
二、前端功能拓展
1. 注册/登录页面：新增注册和登录表单页面，支持用户名+密码的注册登录流程。
2. 登录状态管理：在前端实现Token存储（localStorage），并在每次API请求时自动携带 Authorization 头。
3. 留言展示优化：每条留言显示提交者的用户名，并在用户已登录时高亮显示自己的留言。
4. 删除权限控制：当登录用户角色为管理员时，每条留言旁显示删除按钮，点击后调用 DELETE 
/api/messages/:id 接口删除对应留言。
5. 登录状态UI反馈：页面右上角显示当前登录用户名，并提供退出登录按钮；未登录时显示“登录/注册”入口。
6. 注册/登录跳转：未登录用户尝试提交留言时，自动跳转到登录页面。
三、数据迁移与初始化
1. 编写数据库迁移脚本（migrate.sql），包含 users 表的创建和 messages 表的结构变更（新增 user_id 外
键）。
2. 使用ConfigMap挂载迁移脚本，通过Init Container或手动执行完成数据库迁移。

方向四：自选项目完整部署
 
📋 任务概述
 
本项任务要求选择自己独立部署个人的完整项目（如课程设计、毕业设计、个人作品等），完成代码到容器化再到
Kubernetes部署的全流程。
最终目标：具备将任意前后端分离项目容器化并部署到Kubernetes集群的独立实操能力，实现“自己的项目上云”。
🎯 任务要求
 
一、项目要求
1. 项目必须为前后端分离架构（前端 + 后端API），技术栈不限（Node.js/Java/Python/Go等均可）。
2. 项目后端需涉及数据持久化存储（MySQL/PostgreSQL/MongoDB等任意数据库）。
3. 项目应具备完整的功能闭环（如博客系统、电商系统、任务管理、内容管理等），纯静态页面或仅单页展示
类项目不适用。
4. 项目代码为学生本人独立开发，或团队合作项目中承担主要开发角色。
二、容器化要求
1. 为前端和后端分别编写Dockerfile，要求采用多阶段构建优化镜像体积。
2. 编写.dockerignore文件，排除node_modules、日志文件等构建上下文中的冗余内容。
3. 使用非root用户运行容器进程，确保容器安全性。
4. 为后端服务配置健康检查（HEALTHCHECK或HTTP探针）。
5. 使用docker build分别构建前端和后端镜像，镜像命名规范为：<项目名>-frontend:<版本> 和 <项目名>-
backend:<版本>。
三、Kubernetes部署要求
1. 创建独立的Namespace（命名规范：<项目名>-prod）。
2. 为后端服务创建Deployment（副本数 ≥ 2），配置资源限制（requests/limits）和存活探针/就绪探针。
3. 为前端服务创建Deployment（副本数 ≥ 2），配置资源限制。

7. 使用ConfigMap管理应用配置文件（环境变量、Nginx配置等）。
8. 可选但推荐：使用Ingress配置统一访问入口（域名/路径路由）。
四、部署与验证
1. 按正确顺序部署资源：存储 → 数据库StatefulSet → 后端Deployment → 前端Deployment → Ingress（如使
用）。
2. 验证所有Pod状态均为 Running，所有PVC状态为 Bound。
3. 验证前端页面可正常访问，后端API健康检查返回正常。
4. 验证项目核心功能完整可用（如增删改查、用户认证等）。
5. 验证数据持久化：提交数据后删除数据库Pod，重建后数据完整保留。
